# 扩展文档

本文档讲的是“后面怎么扩”，让项目可以从一个登录示例慢慢变成完整的电商接口自动化平台。

## 1. 建议的扩展方向

### 1.1 按业务模块扩展

电商平台一般可以按下面方式扩展：

- 用户模块
- 商品模块
- 购物车模块
- 订单模块
- 支付模块
- 营销模块
- 库存模块
- 售后模块

建议每个模块都按下面结构扩展：

- `api/xxx_api.py`
- `data/xxx/`
- `testcases/test_xxx.py`

### 1.2 按环境扩展

现在项目里有：

- [uat.yaml](/Users/xueguanglai/Desktop/work/CSJKtest/config/env/uat.yaml)

后面可以继续加：

- `config/env/test.yaml`
- `config/env/pre.yaml`
- `config/env/prod.yaml`

不同环境只改配置，不改用例代码。

### 1.3 按账号角色扩展

电商项目很少只有一个账号，所以建议按角色管理账号。

例如：

- 订单账号
- 商品账号
- 售后账号
- 财务账号

统一放在：

- [account.yaml](/Users/xueguanglai/Desktop/work/CSJKtest/data/account/account.yaml)

后面需要扩，只要新增一个角色，再补一个对应夹具或直接复用 `role_login` 即可。

### 1.4 按断言能力扩展

当前项目断言比较简单，后面可以继续加：

- 返回字段断言
- 数据库断言
- Redis 断言
- 消息队列断言
- 订单状态流转断言

建议把这些能力继续放到 `common` 目录，不要直接塞进测试用例里。

## 2. 推荐的新增模块方式

### 方案一：最简单方式

适合团队刚开始用框架时。

做法：

- 一个接口对应一个数据文件
- 一个业务模块对应一个测试文件
- 用例里直接读取 yaml 数据执行

优点：

- 很容易理解
- 新人容易接手

### 方案二：接口对象化

适合项目后期接口越来越多时。

做法：

- 在 `api` 层封装业务方法
- 例如：`login_by_password()`、`create_order()`、`cancel_order()`

优点：

- 代码复用更好
- 业务语义更清楚

注意：

不要一开始就封装太重，容易把简单项目写复杂。

## 3. 登录态和接口关联怎么扩

很多电商接口不是单独跑的，而是有上下游关系。

例如：

- 先登录
- 再查询用户信息
- 再下单
- 再支付

建议后面这样扩：

### 第一步：提取 token

在响应里拿到登录 token。

### 第二步：把 token 放到公共头里

后续请求统一带上 token。

### 第三步：做接口关联

例如：

- 创建订单后提取 `orderId`
- 支付接口使用 `orderId`
- 取消订单再使用 `orderId`

## 4. 参数化怎么扩

当前示例已经用了 `pytest.mark.parametrize`。

后面你可以继续扩成：

- 正常数据
- 异常数据
- 边界数据
- 权限数据

建议一个 yaml 文件里放多条 case。

例如：

```yaml
- title: 正常登录
  method: post
  url: /xxx
  body: {}
  expected_status_code: 200

- title: 密码错误
  method: post
  url: /xxx
  body: {}
  expected_status_code: 200
```

## 7. CI 持续集成怎么扩

后面接 Jenkins、GitLab CI 或 GitHub Actions 时，建议流程如下：

1. 拉代码
2. 安装依赖
3. 执行 pytest
4. 生成 allure 结果
5. 发布报告
6. 失败后通知群消息

## 8. 框架维护原则

- 先保证能跑，再考虑高级封装
- 先保证大家看得懂，再考虑技巧
- 相同逻辑抽公共方法
- 不要过度设计
- 配置和数据尽量分离
- 用例只保留测试意图

## 10. 推荐的电商落地顺序

建议团队按下面顺序真正落地：

1. 先把登录接口跑通，并确认 `token` 提取规则
2. 再接商品查询，提取 `skuId`
3. 再接购物车接口
4. 再接创建订单接口，提取 `orderId`
5. 最后补支付、退款、售后等模块

## 11. 当前多账号方案怎么扩

当前已经支持：

- `order_user`
- `product_user`
- `refund_user`

如果你后面还要加别的业务，例如库存或营销，建议继续这样做：

1. 在 `data/account/account.yaml` 新增账号角色
2. 在 `conftest.py` 新增对应登录夹具
3. 在 `testcases` 的模块用例里绑定对应夹具

例如新增库存账号：

```yaml
stock_user:
  areaCode: "86"
  phone: "请替换成库存账号"
  password: "请替换成库存密码"
  sliderCode: ""
  signKey: "xxx"
  loginType: 2
```

然后新增：

```python
@pytest.fixture(scope="session")
def stock_login_token(role_login):
    return role_login("stock_user")
```
