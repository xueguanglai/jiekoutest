# 知识库说明

这份知识库不是讲框架怎么写，而是讲团队成员在做接口自动化时，脑子里要有的基础规则。

## 1. 什么是接口自动化

接口自动化就是用程序代替人工去发送请求、校验结果。

它的核心价值有三个：

- 回归更快
- 执行更稳定
- 结果更容易留痕

## 2. 为什么这个项目选 pytest + requests + allure

### pytest

作用：

- 负责执行测试
- 负责参数化
- 负责夹具管理
- 负责筛选用例

优点：

- 学习成本低
- 社区成熟
- 适合团队协作

### requests

作用：

- 负责发送 HTTP 请求

优点：

- 语法简单
- 可读性好
- 很适合接口测试

### allure

作用：

- 负责生成测试报告

优点：

- 报告清晰
- 适合给测试、开发、产品一起看
- 能看到步骤、请求和响应

## 3. 接口自动化项目里最常见的组成

通常会有这几块：

- 配置管理
- 公共请求方法
- 测试数据管理
- 测试用例管理
- 报告管理

这个项目就是按这个思路搭的。

## 4. 为什么要把数据和代码分开

如果把测试数据直接写死在代码里，会有几个问题：

- 改数据要改代码
- 不方便非开发同学维护
- 一旦 case 多了会很乱

所以项目里把测试数据放到了 `yaml` 文件里。

## 5. 为什么要做环境隔离

电商项目一般至少会有这些环境：

- 测试环境
- UAT 环境
- 预发环境
- 生产环境

如果把环境地址写死在代码里，切换环境就会很麻烦。

所以项目里用了 `config/env` 来管理环境。

## 6. 什么是好的自动化用例

好的自动化用例，通常有这些特点：

- 目标明确
- 断言清楚
- 数据独立
- 可重复执行
- 失败后容易排查

不建议写成这种样子：

- 一个用例验证很多业务点
- 一个用例依赖很多前置状态
- 一个用例失败后看不出哪里错

## 7. 电商接口常见断言点

做电商平台测试时，可以重点关注这些断言点：

- HTTP 状态码
- 业务状态码
- 返回提示信息
- 返回字段是否完整
- 订单状态是否正确
- 库存是否正确扣减
- 金额是否正确
- 权限是否正确

## 8. 常见测试数据类型

一般可以分成四类：

- 正常数据
- 异常数据
- 边界数据
- 业务规则数据

例如登录接口：

- 正常手机号 + 正常密码
- 正常手机号 + 错误密码
- 空手机号
- 空密码
- 非法区号

## 9. 接口自动化和 UI 自动化的区别

接口自动化更关注：

- 数据正确
- 接口稳定
- 业务链路

UI 自动化更关注：

- 页面展示
- 页面操作流程
- 前端交互

接口自动化一般更适合做大量回归。

## 10. 团队落地建议

如果团队刚开始做自动化，建议分三步走：

1. 先把核心接口跑起来，比如登录、商品、下单、支付
2. 再把常用场景串起来，比如下单主流程
3. 最后再接持续集成和通知

## 12. 当前框架的设计原则

- 能少一层封装，就少一层封装
- 能直接看懂，就不要故意写复杂
- 通用能力放公共层

- 先满足团队使用，再做高级设计

## 13. 当前增强版新增知识点

### 13.1 什么是上下文变量

上下文变量就是前一个接口返回的数据，给后一个接口继续使用。

例如：

- 登录接口返回 `token`
- 商品接口返回 `skuId`
- 订单接口返回 `orderId`

这些数据都可以放到上下文里统一管理。

### 13.2 什么是占位符替换

占位符替换就是把测试数据里的变量，在真正发请求前替换成真实值。

例如：

```yaml
skuId: "{{skuId}}"
```

执行时，框架会自动去上下文里取值。

### 13.3 什么是提取规则

提取规则就是告诉框架，要从响应里拿哪个字段。

例如：

```yaml
extract:
  orderId: data.orderId
```

意思是：

- 从响应的 `data.orderId` 拿值
- 保存到上下文变量 `orderId`
