# 团队维护规范

这份文档是给团队长期使用的，重点不是“怎么写得高级”，而是“怎么写得稳定、统一、好维护”。

## 1. 目录规范

- `api` 只放接口路径和接口方法
- `common` 只放公共工具
- `config` 只放项目配置和环境配置
- `data` 只放测试数据
- `testcases` 只放测试用例
- `docs` 只放说明文档

## 2. 命名规范

- 接口文件：`xxx_api.py`
- 测试文件：`test_xxx.py`
- 数据文件：`xxx_case.yaml`
- 用例标题：用中文写清楚测试意图

例如：

- `test_order.py`
- `order_case.yaml`
- `创建订单_正常下单`

## 3. 用例编写规范

- 一条用例只验证一个核心目标
- 用例里不要写太多业务细节
- 复杂逻辑放到 `common` 或 `api`
- 每条 case 要有清楚的断言

## 4. 数据管理规范

- 正常数据、异常数据、边界数据尽量放一起管理
- 相同接口放同一个 yaml 文件
- 动态值使用占位符，例如 `{{token}}`
- 上下游接口关联值统一放上下文，不要手写复制

## 5. 断言规范

最少要断言：

- HTTP 状态码
- 业务状态
- 核心字段

关键接口建议再补：

- 金额字段
- 订单状态
- 库存结果
- 权限控制结果

## 6. 登录态规范

- 登录接口负责提取 `token`
- 后续接口自动带 `Authorization`
- 如果项目不是 `Authorization`，就在 `project.yaml` 改 `token_header_name`

## 7. 接口关联规范

常见关联字段：

- `token`
- `userId`
- `skuId`
- `cartId`
- `orderId`
- `addressId`

建议统一在 yaml 中写：

```yaml
extract:
  orderId: data.orderId
```

## 8. 提交前检查规范

提交前至少检查这些：

1. 新增用例是否能收集
2. yaml 格式是否正确
3. 断言是否足够清楚
4. 是否写了无意义的硬编码
5. 文档是否同步更新

## 9. 长期维护建议

- 每个模块指定负责人
- 每次新增模块都补模板数据
- 每周清理失效用例
- 每月复盘断言是否足够覆盖核心风险
