# 接口返回结构约定

这份文档是给团队统一写法用的，避免每个人按自己的理解写断言和提取规则。

## 1. 当前项目的默认返回结构假设

当前项目优先按下面结构处理：

```json
{
  "code": "200",
  "data": {},
  "message": "操作成功"
}
```

## 2. 登录接口约定

当前登录接口真实返回里，已经确认的字段有：

- `data.jwtToken`
- `data.memberId`
- `data.phone`

所以登录提取规则统一按下面写：

```yaml
extract:
  jwtToken: data.jwtToken
  memberId: data.memberId
  phone: data.phone
```

## 3. 订单相关接口约定

当前订单链路里，已经用到的关联字段有：

- `orderId`

默认按下面路径提取：

```yaml
extract:
  orderId: data.orderId
```

如果真实返回不是这个路径，就按真实报文改，不要硬套。

## 4. 业务码约定

现在项目里常见业务码写法有两种：

- `0`
- `200`

所以 yaml 里建议写成：

```yaml
expected_business_codes:
  - 0
  - 200
```

如果某个接口明确只会返回 `200`，那就只写 `200`。

## 5. 字段断言约定

### 5.1 最低要求

每条接口最少断言：

- HTTP 状态码
- 业务码
- 核心字段是否存在

### 5.2 常用写法

字段存在断言：

```yaml
expected_keys:
  - code
  - data
```

字段值断言：

```yaml
expected_values:
  message: 操作成功
```

## 6. 关联变量约定

上下游接口关联时，统一使用 `{{变量名}}`。

例如：

```yaml
body:
  orderId: "{{orderId}}"
```

不要手工复制返回值到下一个接口里。

## 7. 如果接口返回结构不一致怎么办

如果某个新模块返回结构和当前约定不一样，按下面处理：

1. 先保留当前框架不动
2. 只改该模块自己的 yaml
3. 如有必要，再补公共断言能力

不要为了一个特殊接口把全局逻辑改乱。
