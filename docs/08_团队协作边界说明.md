# 团队协作边界说明

这份文档是为了避免多人一起改项目时互相影响。

## 1. 哪些地方可以放心改

业务测试同学优先改这些目录：

- `data/`
- `testcases/`
- `api/`

这些目录主要承载业务差异，适合模块负责人维护。

## 2. 哪些地方不要随便改

下面这些是公共层，建议由框架负责人维护：

- `common/`
- `conftest.py`
- `config/project.yaml`
- `pytest.ini`

如果一定要改，先在团队里说明原因。

## 3. 推荐的分工方式

建议按模块分负责人：

- 订单负责人
- 商品负责人
- 售后负责人
- 库存负责人

每个负责人主要维护：

- 模块数据文件
- 模块测试文件
- 模块接口路径文件

## 4. 新增功能时谁来改什么

### 4.1 新增业务接口

业务同学改：

- `data/模块/`
- `api/模块_api.py`
- `testcases/test_模块.py`

### 4.2 新增公共能力

框架负责人改：

- `common/`
- `conftest.py`

## 5. 提交前最低要求

每次提交前至少检查：

1. 当前模块用例能收集
2. yaml 格式正确
3. 占位账号没有误提交成真实敏感信息
4. 报告能正常生成
5. 文档是否需要同步

## 6. 敏感信息管理建议

- 真实账号密码不要随意扩散
- 如果后面要做更规范，建议改成环境变量或独立私有配置
- 不建议把生产账号直接写进仓库
